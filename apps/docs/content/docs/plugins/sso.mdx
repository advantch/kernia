---
title: "SSO"
description: "Add enterprise SSO with OIDC and SAML providers."
---

The SSO package stores enterprise identity providers, verifies domains, starts OIDC or SAML sign-in, handles callbacks, and provisions users or organizations according to configured policy.

## Installation

~~~bash
uv add kernia-sso
~~~

## Import path

~~~python
from kernia_sso import sso
~~~

## Server configuration

~~~python title="auth.py"
import os

from kernia import KerniaOptions
from kernia.auth import init
from kernia_sso import sso

from .db import adapter

auth = init(KerniaOptions(
    database=adapter,
    secret=os.environ["KERNIA_SECRET"],
    base_url=os.environ["KERNIA_BASE_URL"],
    base_path="/api/auth",
    plugins=(
        sso(),
    ),
))
~~~

## Schema impact

Adds SSO provider and domain tables from `kernia_sso.schema`. Run migrations before exposing admin provider setup.

## Behavior and options

- Protect provider CRUD behind admin routes.
- Domain verification prevents arbitrary tenants from claiming a domain.
- SAML security checks must cover signatures, audience, recipient, timestamps, request IDs, and replay prevention.

## SAML validation modes

The plugin exposes two validation modes via `advanced["sso"]["saml_validation"]`:

- `"strict"` (default) — runs `python3-saml`'s full XML-DSIG verifier against
  the IdP's certificate. This is the production-correct setting and works
  out-of-the-box against the bundled `MockSAMLIdP` test fixture (the fixture
  canonicalizes via `lxml` exclusive C14N, so its signature value reconstructs
  byte-for-byte during verification).
- `"permissive"` — verifies issuer, audience, NotBefore/NotOnOrAfter,
  InResponseTo, Status, the Reference URI, and that the embedded
  `X509Certificate` matches the provider's configured PEM, but does NOT verify
  the XML-DSIG signature value. Use this only for legacy IdPs whose
  canonicalization isn't libxml2-compatible.

Switch in `auth.py`:

~~~python title="auth.py"
auth = init(KerniaOptions(
    ...,
    advanced={"sso": {"saml_validation": "strict"}},
))
~~~

## Test and demo coverage

Use `MockSAMLIdP` (strict-mode compatible) and `MockIdP` for OIDC. Tests
cover provider CRUD, domain verification, OIDC callback, SAML metadata, SAML
ACS, replay rejection, timestamp validation, IdP-initiated flows when
enabled, and strict-mode signature rejection with a wrong cert.
